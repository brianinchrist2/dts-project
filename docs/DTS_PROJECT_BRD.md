# DTS-Project 業務需求說明書 (BRD)

## 1. 項目目標
本項目旨在構建一個高性能、高可靠的金融交易接口中台服務。系統通過對接 Docker 容器內的 MetaTrader 5 (MT5) 終端，對外提供標準化的行情數據與交易執行 RESTful 接口。

**核心目標**：提供穩定、低延遲的 API 服務。
**輔助目標**：提供一套輕量級的可視化監控界面。
**項目邊界**：本項目不包含交易策略邏輯。具體的「專家交易系統（EA）」由獨立的項目負責，通過本項目提供的 API 接口進行交互。

## 2. 角色定義
- **外部專家系統 (External Expert System)**：本項目的核心調用者。通過 API 獲取數據、計算策略並下達交易指令。
- **監控人員/用戶**：通過輔助前端界面觀察系統運行狀態、實時行情及賬戶概況，不參與交易決策。

## 3. 功能需求

### 3.1 核心服務：數據 API (Data Service API)
- **實時行情推送/拉取**：
  - 支持批量獲取白名單品種的最新 Bid/Ask/Last 報價。
  - 提供各品種最近 20 條 Tick 的序列數據，滿足外部系統對盤口微觀結構的分析需求。
- **多維度 K 線服務**：
  - 維護 M1 級原始 K 線數據庫。
  - 提供動態合成服務：根據外部調用參數，實時合成 M5、H1、D1 等高週期 OHLCV 數據。
- **技術指標中台**：
  - 接口需集成指標計算邏輯，支持外部系統直接通過 API 獲取計算好的技術指標結果（如 RSI, EMA 等），減少外部系統的計算負擔。

### 3.2 核心服務：交易 API (Trading Service API)
作為外部專家系統的執行單元，需提供極其穩健的指令通道：
- **全功能交易指令**：涵蓋市價單、限價單（Limit）、停損單（Stop）的增、刪、改、查。
- **資產狀態同步**：實時提供精確的賬戶餘額、淨值、持倉盈虧及保證金水平。
- **歷史審計接口**：支持外部系統追溯歷史成交記錄，用於策略盈虧分析。

### 3.3 輔助系統：監控界面 (Monitoring UI)
- **定位**：純展示、非交互。用於驗證 API 數據的正確性及系統運行的直觀觀察。
- **監控內容**：
  - **行情監視器**：主要品種的實時價格動態。
  - **資產儀表盤**：賬戶核心資金曲線及當前風險率。
  - **狀態指示燈**：顯示與 MT5 容器、數據庫、API 服務的健康度。

## 4. 項目目錄結構
- `dts-project/` (總目錄)
  - `dts-service/` (接口服務層 - 項目核心)
    - 實現高並發、標準化的 RESTful API。
    - 負責指標合成邏輯及外部訪問的安全鑒權。
  - `dts-expert/` (核心連接引擎)
    - 直接對接 MT5 終端的底層進程。
    - 負責「數據抓取」與「指令下發」的管道化操作，確保與 MT5 容器通信的穩定性。
  - `dts-face/` (輔助監控前端)
    - 基於 Web 技術的唯讀看板，通過 API 獲取數據進行可視化。

## 5. 部署與運行架構
- **宿主機 (Linux VPS)**：運行 `dts-service`、`dts-expert` 進程及數據庫。
- **容器層 (Docker)**：運行容器化的 MT5 終端（通過 Wine 模擬）。
- **通信模式**：宿主機服務通過網絡端口（如 18888）與 Docker 內的 MT5 插件進行雙向數據交換。

## 6. 系統配置需求
- **品種規則**：預設各品種的精度、手數步長、最小點值等。
- **服務配置**：定義 API 的訪問權限、心跳頻率、數據保留週期等參數。

## 7. 非功能性需求

### 7.1 服務性能 (核心關注)
- **高可用性**：支持 7x24 小時運行，具備進程級故障自愈能力。
- **吞吐量**：API 需支持多個外部實例同時調用的並發壓力。
- **低延遲**：從 MT5 報價產生到 API 響應的物理延遲需降至最低。

### 7.2 安全與穩定性
- **指令鑒權**：交易 API 必須具備嚴格的身份驗證，防止未授權訪問。
- **環境隔離**：確保 Docker 容器的重啟不影響宿主機服務的持久化狀態。

## 8. 業務規則約束
- **單一事實來源**：所有數據必須以 MT5 交易服務器為準，系統本地存儲僅作為緩存和索引。
- **指令反饋機制**：交易 API 必須提供明確的異步或同步反饋，確保外部系統獲知訂單的最終執行狀態。
